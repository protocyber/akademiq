# Audit: `apps/web` vs Next.js 16 fresh install

**Date:** 2026-06-18  
**Scope:** Compare `apps/web` config/structure against a pristine `pnpm create next-app@latest my-app --yes` (Next 16.2.9) to identify drift, missing defaults, and stale v3 patterns after the v14 → v16 upgrade.  
**Reference install:** `/tmp/my-app` generated from `create-next-app@latest`.

---

## 1. Files that MUST change to match Next 16 defaults

### `next.config.js` → `next.config.ts` (ESM)

| Aspect | Fresh install | `apps/web` | Action |
|---|---|---|---|
| Extension | `.ts` | `.js` (CJS `module.exports`) | **Migrated to `.ts`** ✅ |
| Syntax | `import type { NextConfig }` + `export default` | `/** @type {import("next").NextConfig} */` + `module.exports` | **Migrated** ✅ |

The `.js` CJS style still works but Next 16 ships with `.ts` by default.
Migrating gives full autocomplete and catches typos at config load time.

### `eslint.config.mjs` — structure upgrade

| Aspect | Fresh install | Old `apps/web` | **Migrated** |
|---|---|---|---|
| Import style | `defineConfig` + `globalIgnores` from `"eslint/config"` | Spread `...next` from default export | ✅ |
| Granular imports | `eslint-config-next/core-web-vitals` + `/typescript` | Single `eslint-config-next` | ✅ |
| `@typescript-eslint/no-unused-vars` | `warn` (via typescript preset) | Implicit `warn` (via default export) | **Promoted to `error`** with `_` ignore patterns ✅ |
| Custom rules | Not present | `react/forbid-elements`, `react-hooks/set-state-in-effect: warn` | **Preserved** ✅ |

### `@typescript-eslint/no-unused-vars` — promoted to `error`

The new granular config activates `typescript-eslint` recommended rules which
include `no-unused-vars`. The project chose to promote it from `warn` to
`error` with underscore-prefixed identifier exemptions:

```js
"@typescript-eslint/no-unused-vars": [
  "error",
  {
    argsIgnorePattern: "^_",
    varsIgnorePattern: "^_",
    caughtErrorsIgnorePattern: "^_",
  },
],
```

This catches dead imports/variables at lint time (previously silent). The `_`
patterns allow intentional destructuring-strip idiom:
`const { remember_device: _rememberDevice, ...input } = values`.

## 2. Files that SHOULD stay different (by design)

### Tailwind v3 (not v4)

| Aspect | Fresh install (Tailwind **v4**) | `apps/web` (Tailwind **v3**) |
|---|---|---|
| `tailwindcss` version | `^4` (4.3.1) | `3.4.16` |
| PostCSS plugin | `@tailwindcss/postcss` | `tailwindcss` + `autoprefixer` |
| Config file | **None** — config via CSS `@theme` directive | `tailwind.config.ts` |
| CSS directives | `@import "tailwindcss";` | `@tailwind base/components/utilities;` |

**Decision: Keep Tailwind v3.** The project uses shadcn/ui which relies on
CSS-variable-based theming (`hsl(var(--primary))`, `tailwindcss-animate` plugin,
`darkMode: ["class"]`). Migrating to Tailwind v4 requires rewriting
`tailwind.config.ts` → `@theme` in `globals.css` and updating every shadcn/ui
component. This is a large, separate effort unrelated to the Next.js upgrade.

> Next 16 does NOT require Tailwind v4. Both v3 and v4 are supported.

### `postcss.config.js` (CJS)

Kept as CJS because it pairs with Tailwind v3's `tailwindcss` plugin
(v4 would switch to `@tailwindcss/postcss` and `.mjs`).

### `src/` directory layout

Fresh install puts `app/` at the project root; `apps/web` uses `src/app/`.
Both are valid. The `src/` layout was a deliberate early choice and requires
no change. `tsconfig.json` paths correctly reflect it (`"@/*": ["./src/*"]`).

### Custom ESLint rules preserved

| Rule | Value | Rationale |
|---|---|---|
| `react/forbid-elements` | `error` (exemptions for UI primitives, layouts, tests) | shadcn/ui-only UI policy |
| `react-hooks/set-state-in-effect` | `warn` | React Compiler rule; codebase predates it; refactor tracked separately |
| `react-hooks/incompatible-library` | `off` | Not applicable |

## 4. Fixes applied during audit

### 38 new lint errors resolved (typescript-eslint recommended rules)

The granular `eslint-config-next/typescript` import activates
`typescript-eslint` recommended rules which were NOT loaded by the old
spread-based `...next` config. This surfaced 38 new errors across 7 files:

| File | Count | Fix |
|---|---|---|
| `tailwind.config.ts` | 1 (`no-require-imports`) | `require()` → ESM `import tailwindcssAnimate` |
| `src/components/icons/google-icon.tsx` | 1 (`no-empty-object-type`) | Empty interface → `type` alias |
| `src/components/features/academic-config/status-confirm-dialog.tsx` | 1 (`no-explicit-any`) | `catch (err: any)` → `unknown` + type narrowing |
| `src/app/settings/academic/years/page.tsx` | 4 (`no-explicit-any`) | `as any` → proper union types (`AcademicYearStatus`/`AcademicTermStatus`), added type exports to schemas |
| `__tests__/academic-scope.test.tsx` | 17 (`no-explicit-any`) | `as any` → `as unknown as never` (mock pattern) |
| `__tests__/apply-server-field-errors.test.ts` | 4 (`no-explicit-any`) | `form as any` → `form as unknown as UseFormReturn<FieldValues>` |
| `__tests__/sidebar-nav-visibility.test.tsx` | 10 (`no-explicit-any`) | `as any` → `as unknown as never` (mock pattern) |

### 33 unused imports/variables removed

Dead imports removed across 16 files. Full list in the commit diff.
`@typescript-eslint/no-unused-vars` promoted from `warn` to `error`
in `eslint.config.mjs` to catch future dead code at lint time.

## 6. Pre-existing issues (not addressed in this audit)

- **12 test failures** in `academic-scope.test.tsx` (7) and `sidebar-nav-visibility.test.tsx` (5) — caused by a missing `useTerms` export in the mock factory for `@/lib/query/queries/use-academic-config`. - **`react-hooks/set-state-in-effect` warnings (36)** — React Compiler rule flagging `setState` inside `useEffect`. Many are legitimate sync patterns. Tracked separately; downgraded to `warn` intentionally.
